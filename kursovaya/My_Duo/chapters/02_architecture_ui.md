## 1.4 Проектирование карты экранов (User Flow) и структуры команд

Эффективность пользовательского интерфейса Telegram-бота напрямую зависит от правильного проектирования сценариев переходов между экранами (User Flow). Администратор системы должен иметь возможность управлять всеми функциями мониторинга в один-два клика, не теряя текущего контекста.

В основе интерфейса проектируемой системы лежит концепция «Единого интерактивного окна». Вместо генерации цепочки новых сообщений, бот при получении команд от пользователя динамически редактирует исходное приветственное сообщение, заменяя в нём текст и набор inline-кнопок. Это позволяет сгруппировать все административные функции в логические экраны:
1.  **Главный экран (Главное меню)**. Отображается при старте бота или нажатии кнопки «Назад в меню». Содержит краткий статус работы системы и базовые кнопки навигации: «Управление каналами», «Добавить канал», «Управление службами», «Сформировать дайджест», «Справка».
2.  **Экран управления каналами**. Представляет собой интерактивный список всех подключённых к мониторингу Telegram-каналов с поддержкой постраничной пагинации (если каналов больше 5). Каждый канал в списке кликабелен.
3.  **Экран карточки канала**. Открывается при выборе конкретного канала. Выводит информацию о количестве собранных постов, дате последней проверки и предлагает кнопки «Удалить из мониторинга» или «Вернуться к списку».
4.  **Экран добавления нового канала**. Переводит интерфейс бота в режим ожидания текстового ввода. Бот просит пользователя прислать ссылку или юзернейм канала, проверяет его валидность и записывает в базу данных.
5.  **Экран ручного запроса дайджеста**. Позволяет сгенерировать сводный новостной отчёт по выбранным регионам за последние 24 часа.

## 1.5 Логика безопасности и разграничения прав доступа

Поскольку разрабатываемый бот предоставляет мощные инструменты управления системой сбора данных (добавление/удаление источников мониторинга, остановка системных служб, доступ к полному архиву новостей), вопросы информационной безопасности и ограничения доступа к интерфейсу имеют первостепенное значение.

Традиционные схемы авторизации через ввод логина и пароля внутри мессенджера крайне неудобны и небезопасны (историю сообщений можно перехватить или прочесть с экрана). В проектируемой системе реализован наиболее надёжный метод авторизации — жесткая фильтрация на основе уникального идентификатора чата (chat_id) Telegram.

Каждое входящее обновление (Update), отправляемое серверами Telegram на сервер бота, содержит уникальный числовой идентификатор отправителя `from.id` (или `chat.id`). В процессе инициализации бота из конфигурационного файла `.env` считывается список доверенных администраторов `ADMIN_IDS` (формат строки, содержащей числа через запятую).

Логика работы фильтра безопасности строится следующим образом:
1.  При получении любого обновления от Telegram API (будь то текстовое сообщение или клик по inline-кнопке) маршрутизатор извлекает `chat_id` пользователя.
2.  Производится проверка: входит ли данный `chat_id` в список разрешённых администраторов.
3.  Если идентификатор отсутствует в списке, бот выполняет следующие действия:
    *   Полностью игнорирует запрос (не выполняет никаких действий в базе данных).
    *   Отправляет пользователю вежливое уведомление об отсутствии прав доступа.
    *   Записывает попытку несанкционированного доступа в системный лог `loguru` с указанием `username` и `chat_id` нарушителя для последующего анализа.
4.  Если проверка пройдена, запрос передаётся дальше в модуль маршрутизации `UpdateProcessor`.

Такой подход полностью исключает возможность взлома интерфейса методом перебора паролей (Brute Force) или использования уязвимостей в логике сессий, так как подделать уникальный `chat_id` отправителя в рамках защищённого протокола MTProto платформы Telegram технически невозможно.

## 1.6 Спецификация хранения данных пользователя и подписок

Для обеспечения бесперебойной работы интерфейса бота, хранения информации о подключённых каналах и пользовательских подписках спроектирована реляционная база данных на основе СУБД SQLite.

Вся информация хранится в файле `news_monitor.db`. Для поддержки асинхронного пользовательского интерфейса критически важно исключить любые блокировки таблиц при одновременном чтении данных ботом и записи новых постов модулем сбора реального времени. Для этого СУБД переведена в высокопроизводительный режим работы **WAL (Write-Ahead Logging)**, а также установлена синхронизация данных в режиме **NORMAL**.

Ниже представлена детальная спецификация основных таблиц базы данных:

1.  **Таблица channel_checks (Каналы мониторинга)**. Хранит информацию об источниках сбора новостей, отображаемых в интерфейсе управления.
    *   `channel_username` (TEXT, PRIMARY KEY) — юзернейм канала (например, `@news_fefu`).
    *   `last_check_time` (TIMESTAMP) — время последней проверки постов.
    *   `is_active` (INTEGER) — флаг активности мониторинга (1 — активен, 0 — приостановлен).
2.  **Таблица messages (Собраные новости)**. Используется ботом для выборки контента и генерации дайджестов по запросу пользователя.
    *   `id` (TEXT, PRIMARY KEY) — уникальный составной идентификатор поста.
    *   `channel_username` (TEXT) — юзернейм источника публикации.
    *   `text` (TEXT) — очищенный текст новости.
    *   `views` (INTEGER) — количество просмотров поста.
    *   `forwards` (INTEGER) — количество форвардов (поделившихся).
    *   `media_group_id` (TEXT) — идентификатор медиагруппы (альбома).
    *   `created_at` (TIMESTAMP) — дата публикации новости.
3.  **Таблица regions (Регионы)**. Содержит справочник поддерживаемых регионов для настройки подписок.
    *   `name` (TEXT, PRIMARY KEY) — название региона (например, «Владивосток»).
    *   `emoji` (TEXT) — графический символ для визуализации в кнопочном меню (например, «📌»).
4.  **Таблица subscriptions (Подписки пользователей)**. Связывает подписчиков с выбранными регионами.
    *   `user_id` (INTEGER) — уникальный chat_id пользователя в Telegram.
    *   `region_name` (TEXT) — имя региона из таблицы `regions`.
    *   Составной первичный ключ: `PRIMARY KEY (user_id, region_name)`.

SQL-скрипт создания базовой структуры данных представлен в листинге 1.1.

Листинг 1.1 — Структура таблиц базы данных SQLite

```sql
CREATE TABLE IF NOT EXISTS channel_checks (
    channel_username TEXT PRIMARY KEY,
    last_check_time TIMESTAMP,
    is_active INTEGER DEFAULT 1
);

CREATE TABLE IF NOT EXISTS messages (
    id TEXT PRIMARY KEY,
    channel_username TEXT NOT NULL,
    text TEXT,
    views INTEGER DEFAULT 0,
    forwards INTEGER DEFAULT 0,
    media_group_id TEXT,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY(channel_username) REFERENCES channel_checks(channel_username) ON DELETE CASCADE
);

CREATE TABLE IF NOT EXISTS regions (
    name TEXT PRIMARY KEY,
    emoji TEXT NOT NULL
);

CREATE TABLE IF NOT EXISTS subscriptions (
    user_id INTEGER,
    region_name TEXT,
    PRIMARY KEY (user_id, region_name),
    FOREIGN KEY(region_name) REFERENCES regions(name) ON DELETE CASCADE
);

CREATE INDEX IF NOT EXISTS idx_messages_channel ON messages(channel_username);
CREATE INDEX IF NOT EXISTS idx_subscriptions_user ON subscriptions(user_id);
```

Для оптимизации быстродействия интерфейса при выборках созданы индексы `idx_messages_channel` и `idx_subscriptions_user`, которые в разы ускоряют формирование региональных списков новостей для пользователей. Настройки производительности базы данных, применяемые при каждом подключении через `aiosqlite`, выглядят следующим образом (листинг 1.2).

Листинг 1.2 — Настройка производительности SQLite

```sql
PRAGMA journal_mode = WAL;
PRAGMA synchronous = NORMAL;
PRAGMA cache_size = 10000;
PRAGMA temp_store = MEMORY;
```

Эти настройки позволяют боту осуществлять мгновенную выборку данных для интерактивного меню даже в моменты, когда сборщик интенсивно записывает сотни постов в секунду.
